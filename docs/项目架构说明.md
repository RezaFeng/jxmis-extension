# jxmis-extension 项目架构说明

本文描述当前 `src/` 架构、六个浏览器 bundle、Chrome 运行环境、深 module interface、adapter seam、消息协议、测试边界和人工验收要求。历史优化动机与落实对照见 [项目架构优化建议](项目架构优化建议.md)。

## 1. 系统概览

项目是一个 Manifest V3 Chrome 扩展。扩展在 JXMIS 页面中识别日报审核、周报填报和周报审核三类场景，由 ISOLATED content runtime 挂载控件并注入相应 page bundle；page bundle 在 MAIN world 调用页面对象和同源 JXMIS 接口；background service worker 负责配置、缓存和 AI 流式请求；popup 负责用户配置。

源码采用 ESM，esbuild 将六个入口打包为固定文件名的 IIFE。`src/` 是唯一 implementation，`dist/` 是唯一加载目录。

## 2. 运行拓扑

```text
popup.html -> popup.js -> chrome.storage.local
                              |
JXMIS MAIN world <-> window.postMessage <-> content.js (ISOLATED)
       |                              |             |
       |                              |             +-> chrome.runtime Port/Message
       |                              |                            |
       |                              +-> 注入单个 page bundle      v
       |                                           background.js (service worker)
       v                                                        |
page-daily-approval.js                                          +-> AI provider
page-batch-work.js
page-weekly-approval.js
       |
       +-> JXMIS DOM / DataTables / WkFormJS / same-origin fetch
```

运行环境不能合并：MAIN world 才能访问 `DataTablesUtil`、`WkFormJS` 和页面 JavaScript 状态；ISOLATED world 才能稳定使用扩展 API；service worker 承担跨域 AI 请求和持久配置。

### 2.1 MAIN 与 ISOLATED 启动

生产 manifest 在 `document_start` 注册两条不同脚本：

- MAIN world 加载 `dist/background.js`。同一 bundle 在 service worker 中注册 background runtime，在页面 MAIN world 中只安装项目负责人 fetch/XHR 覆盖。
- ISOLATED world 加载 `dist/content.js`，启动页面识别、控件和消息桥。

这里复用 `dist/background.js` 文件名是经过 Chromium 扩展测试验证的约束处理：Chromium 会对 MAIN 和 ISOLATED 两个 manifest 项中的同名脚本去重，导致第二个 world 不执行。使用两个不同的现有 bundle 保持六个输出，不增加第七个 bootstrap 文件。

## 3. 六个入口

| 输出 | 源入口 | 运行位置 | 职责 |
|---|---|---|---|
| `dist/background.js` | `src/entries/background.js` | service worker / MAIN world | background runtime 或项目负责人覆盖 bootstrap |
| `dist/content.js` | `src/entries/content.js` | ISOLATED world | 页面 matcher、控件、page bundle 注入、消息和 AI bridge |

## 5. 深 Module Interface

### 5.1 日报审核

`createDailyApprovalAutomation(adapters)` 返回 `{ run }`。调用方只触发一次运行；分页读取、payload 校验、顺序审批、延迟和状态上报都隐藏在 implementation 中。

### 5.2 周报审核

`createWeeklyApprovalAutomation(adapters)` 返回 `{ preview, run }`。implementation 负责候选过滤、预览、批复前详情复查、批复请求和批复后状态复查。

### 5.3 批量报工

page automation 对消息入口暴露 `run`、`runToolbarAction` 和 `handleMessage`；核心编排进一步收敛在 `createBatchWorkRunner(operations).run(mode)` 后。`mode` 只允许：

- `all`：当前周工时与总结保存后，再生成并按条件保存下周计划。
- `summary`：只生成总结并保存当前周。
- `hours`：只更新当前周工时并按变更决定保存。
- `plan`：只填充下周计划，不自动保存。

调用方不需要学习日报匹配、DataTables change store、AI cache 或 WBS 拆分规则，因而该 interface 对测试和页面入口都有较高 leverage。

### 5.4 Content Runtime

`startContentRuntime(adapters)` 负责 automation registry、页面识别、page bundle 加载、控件状态和跨 world 消息路由。`createStatusControl`、`createPageScriptLoader` 和 `createAiBridge` 是 implementation 内部 seam，分别可替换 DOM、扩展 URL 和 Port 行为。

### 5.5 Background Runtime

`registerBackgroundRuntime(adapters)` 安装 Chrome message/port listener，并组合：

- `createConfigStore`：配置和周报总结 cache。
- `createAiClient`：模型列表和 chat completions。
- SSE parser：status、warning、reasoning、chunk、done、error 和无正文校验。

### 5.6 消息协议

`src/shared/protocol.js` 集中 `SOURCES`、`MESSAGE_TYPES`、`AI_PORT_TYPES` 及构造/解析函数。daily/work/weekly、content 和 background 的跨环境消息通过协议常量协作；带 requestId 的 AI/cache 消息会做最小运行时校验，未知消息返回无效结果而不中断 runtime。

## 6. Adapter Seam

只有存在生产与测试两种 adapter 时才建立 seam：

- JXMIS transport：生产使用 same-origin fetch、页面 location 和 localStorage；测试使用 fake transport。
- 页面对象：生产使用 DOM、DataTables 和 `WkFormJS`；浏览器 fixture 提供最小可观察实现。
- 时间：批量审批生产使用真实 timer/random；契约测试使用 fake clock。
- AI：生产使用 fetch、TextDecoder、Chrome Port；测试使用 fake fetch、fake stream 和 fake port。
- 存储：生产使用 `chrome.storage.local`；测试使用内存对象。

业务 endpoint 和状态规则留在 daily/work/weekly implementation 内，不进入通用 transport adapter。

## 7. 关键数据流

### 7.1 页面自动化

1. content runtime 的 matcher 识别当前页面。
2. automation registry 只注入该页面对应的一个 page bundle。
3. content 创建按钮和状态控件。
4. 用户点击后，content 通过 `window.postMessage` 发送 start/action 消息。
5. MAIN world page automation 执行 JXMIS 请求或页面写入，并持续发回 running/progress/done/error。
6. content 更新控件，必要时刷新 DataTables 或页面。

### 7.2 AI 周报总结

1. batch-work page 生成 prompt 和 requestId，向 content 发送 AI request。
2. content 的 AI bridge 建立命名 Port，并隔离 stale request。
3. background 读取 popup 配置，构造 provider-specific request body。
4. background 解析 SSE，将 reasoning/chunk/done/error 发回 content。
5. content 转回 page；batch-work 聚合正文并写入“本周执行情况”。
6. disconnect 会清理活动请求；同一时刻只保留一个活动 Port。

### 7.3 项目负责人覆盖

1. content 从 `chrome.storage.local` 读取 `projectManager`。
2. content 向 MAIN world 发布配置消息。
3. MAIN bootstrap 包装页面 fetch 和 XMLHttpRequest。
4. 只对同源 `/jxpmo` 且已包含 `projectManager` 的 URL/body 替换值，不新增缺失参数，不改外部 URL。

## 8. 业务不变量

- 日报和周报审批保持顺序执行及原有延迟策略。
- 周报审核在批复前复查负责人和待审核状态，批复后再次复查状态。
- `all` 模式先保存当前周执行数据和周报总结，再生成/校验下周计划，最后才可能触发下周保存。
- 缺少 AI 配置时，`all` 模式可以跳过总结继续；`summary` 模式必须失败。
- 缺少 `majorPerson` 只阻止下周自动保存，不回滚已完成的当前周保存。
- `plan` 模式永不自动保存。
- popup 沿用原 storage key 和配置语义。

## 9. 构建与发布

`scripts/build.mjs` 每次先清空输出目录，再由 esbuild 一次打包六个入口并复制 popup 静态资源：

- development：输出到 `dist/`，保留 inline source map。
- test：输出到 `dist-test/`，只在该 manifest 中追加 `http://127.0.0.1/*` 的 content、host 和 web-accessible 权限。
- production：输出到 `dist/`，压缩且不生成 source map。

`scripts/package.mjs` 先执行 production 构建，再把 `dist/` 内容直接压入 `release/jxmis-extension.zip`。ZIP 根目录包含 manifest，不额外嵌套 `dist/`。

## 10. 测试边界

### 10.1 Node 测试

`test/unit/` 验证 protocol、transport、SSE、WBS、日报匹配、周报上下文和总结纯逻辑；`test/integration/` 从深 module interface 验证日报审批、周报审核、四种报工模式、content bridge 和 background runtime。

### 10.2 离线 Chromium

`test/extension.spec.js` 加载 `dist-test/`，本地 fixture server 模拟三类 JXMIS 页面、DataTables、`WkFormJS`、同源接口和 AI SSE。测试断言：

- 扩展和 service worker 成功加载。
- 每页 matcher 只注入对应 page bundle。
- 控件出现，点击消息可观察。
- fake `saveAll()` 被调用。
- AI chunk/done/error 返回页面。
- 不向真实 `jxmis.cyberwing.cn` 发请求。

### 10.3 不自动化的范围

自动化测试不登录真实 JXMIS、不读取账号/Cookie、不执行真实审批或保存。真实系统 DOM、DataTables shape、权限和接口响应仍需人工 UAT。

## 12. 维护原则

- 新业务规则优先放进现有深 module implementation，不扩大入口 interface。
- page-specific DOM 挂载留在对应页面实现，不把差异强塞进共享 status control。
- 新消息先进入 protocol module，再更新发送方和接收方。
- 新 adapter 必须至少有生产与测试两种真实实现，否则先保持普通函数。
- 修改 shared protocol、transport 或 runner 前先做上游影响分析，并按受影响流程扩大测试。
