# jxmis-extension 项目架构说明

本文描述当前 Manifest V3 Chrome 扩展的运行环境、7 个 bundle、经营分析模块、实时数据流和测试边界。

## 1. 系统概览

扩展包含两类业务：原有日报/周报自动化，以及只读经营分析。content runtime 在 JXPMO 顶层页面识别场景并注入对应 MAIN world page bundle；background service worker 负责 AI 请求和升级时清理旧经营 IndexedDB；统一 Options 页承载 AI、项目经理覆盖、经营筛选和预警阈值。

经营分析在项目模块左侧菜单中挂载“经营分析”。点击后只隐藏 `#content-wrapper` 的原生子内容，在同一容器加入 Shadow DOM 报告；切换到其他项目菜单或路由时恢复原节点的 display、ARIA 和菜单 class。

## 2. 运行拓扑

```text
Options tab -> options.js -> chrome.storage.local

JXPMO MAIN world <-> window.postMessage <-> content.js (ISOLATED)
       |                                       +-> 导航、Shadow DOM、状态与下载
       +-> page-daily-approval.js
       +-> page-batch-work.js
       +-> page-weekly-approval.js
       +-> page-business-analytics.js -> same-origin JXPMO fetch

content.js <-> runtime message/Port <-> background.js -> AI provider
```

MAIN world 才能访问 JXPMO 页面对象和同源运行状态；ISOLATED world 使用 Chrome 扩展 API 并隔离报告 UI；service worker 承担跨域 AI。三者通过 `src/shared/protocol.js` 的受限消息协议协作。

### 4.1 领域内核

`src/analytics/` 不访问 DOM、Chrome API 或网络。它负责 schema、枚举、配置/策略版本、日期范围、公式、风险、35 项核心值、相邻周期比较、全部部门聚合和离线 HTML。比率始终先聚合原值再计算：已知分母为 0 时返回 0，依赖来源技术失败时返回 `null`。

### 4.2 页面采集

`createJxpmoAnalyticsData` 隐藏 endpoint、全分页和响应规范化；`createAnalyticsCollector` 隐藏查询取消、并发限制、面板工作集、来源状态和失败重试 descriptor。组织树、项目、当前/上一周期日报可在同一面板内按条件复用；跨年回款是每次正式查询的共享来源，不按项目重复请求；WBS、里程碑和周报按项目受控并发。

### 4.3 Content 编排

`createBusinessAnalyticsController` 连接 navigation、view 和 engine。打开面板时先向 MAIN world 请求范围初始化；同一筛选和周期内的正式查询复用当前页面内存中的项目与日报工作数据，刷新业务页面后重新初始化。失败重试只复用当前页面内存中的规范化结果和失败项。正式全量报告与临时项目选择报告分离，导出永远使用正式报告。`report-view` 只渲染已经计算好的结果，不从 DOM 反解析指标。

### 4.4 无经营持久化

经营分析不创建 IndexedDB，不保存查询缓存、失败 descriptor、报告快照或指标历史。background 启动时调用 `cleanupLegacyAnalyticsDatabase`，幂等删除旧版本遗留的 `cw-business-analytics`；Options 不提供经营数据清理按钮。配置仍由 `chrome.storage.local` 保存。

## 5. 经营分析数据流

1. controller 使用页面运行时读取的 Options 筛选和阈值，并设置默认最近完整自然周；Options 变更由用户刷新业务页面后生效。
2. page bundle 并行获取组织树、按 `currStatus × outsourcing` 组合的项目全分页，以及配置要求的当前周期日报。项目合并后按 `projectId` 去重，在本地应用完整 Options，并要求 `realExeuCost > 0`；需要本期投入时再按 `sum(realHour) > 0` 收窄范围，以 `projectDept` ID 形成有效部门集合。初始化全部成功前部门和查询控件保持禁用。
3. 用户点击查询后，collector 复用范围工作集，首次按需获取上一周期日报、WBS、里程碑、周报和跨年回款；同一面板内切换部门不重复获取项目、当前周期日报或上一周期日报。
4. 日报 normalizer 以原始 `realEndTime` 作为业务日期，缺失时依次兼容 `submissionTime`、`createTime`，并写入内部 `taskDate`；其他 normalizer 对 ID、日期、金额、工时和枚举做 schema 校验。失败来源记录为 `failed`，合法空记录为 `empty`。
5. collector 只保留本期 `sum(realHour) > 0` 的项目作为正式集合；engine 用同一集合计算实时累计、本期和紧邻上期。
6. 接口成功返回的空业务值按 0；已知零分母比率为 0；失败依赖的指标为 `null`，UI 显示“未获取”。部分报告可用当前页面内存中的失败列表执行“仅重试失败项”。
7. “全部部门”由一次实时采集按部门重新分组计算，公共来源不按部门重复请求，项目按 `projectId` 去重。
8. HTML 导出白名单字段并内嵌 CSP、CSS 和本地交互脚本，不允许外部请求。

详细字段和公式见 [经营分析数据字典](经营分析数据字典.md)。

## 6. 关键不变量

- 不保存、配置、输出或日志记录 `JSESSIONID`、Cookie 和 Authorization header。
- 经营接口保持当前用户页面权限，不通过参数扩大可见范围。
- 部门匹配只使用 ID；名称只展示。非当前组织部门仅进入汇总诊断。
- 同一筛选维度内为 OR，不同维度间为 AND；部门最后按 ID 精确过滤。
- 日报人天为 `sum(realHour) / 8`，成本直接汇总日报 `cost`，不使用固定 800 元回退。
- WBS 实际完成只统计 `actualEndTime <= endDate`，未来记录不能穿透历史截止日。
- 周报按日期交集选取；同项目同周期保留最新有效版本，跨周聚合。
- 回款按项目 ID 或唯一合同号关联，不按销售部门分摊。
- 部分结果可查看和导出；技术失败保留为“未获取”，不能伪装为 0。
- 临时选择不覆盖顶部正式统计，也不进入导出。
- AI 只用于周报总结，经营数据不会发送给 AI provider。

## 7. 原有自动化边界

- `createDailyApprovalAutomation` 暴露 `run`，内部保持顺序审批和状态上报。
- `createWeeklyApprovalAutomation` 暴露 `preview/run`，批复前后都复查状态。
- `createBatchWorkRunner.run(mode)` 支持 `all/summary/hours/plan`；`plan` 永不自动保存，`all` 先保存当前周再处理下周。
- `startContentRuntime` 负责 automation registry 和单 page bundle 注入，不让不同业务页面互相加载脚本。

## 9. 真实系统验收与归档

自动化不读取真实账号/Cookie，不执行真实审批、保存或经营双跑。真实接口 shape、权限、部门范围和业务口径必须按 [经营分析 UAT](经营分析UAT.md) 人工验证。

旧 `weekly_report_toolkit_V3.1` 只允许用于只读双跑对账。扩展不存在对旧仓库、Node/Python 流水线、Playwright 抓取、Excel 或 `output/` 的运行时引用。完成 3 部门 × 2 周对账并签字后，才可把旧工具转为只读归档。

## 10. 维护原则

- 修改 shared protocol、engine、collector 或 runtime 前先做上游影响分析。
- endpoint 和响应 shape 留在 page data/normalizer，不扩散到 controller 或 view。
- 指标公式只存在于 `src/analytics/`，页面和 background 不复制公式。
- 新消息先进入 `src/shared/protocol.js`，并同时覆盖发送方、接收方和协议测试。
- 新 adapter 只有在生产与测试均有真实替代实现时建立。
- 高风险改动按受影响执行流扩大单元、集成和浏览器测试。
