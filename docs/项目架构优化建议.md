# jxmis-extension 项目架构优化建议

生成时间：2026-07-04  
范围：`content.js`、`page-batch-work.js`、`page-batch-approve.js`、`page-batch-weekly-approve.js`、`background.js`、`popup.js` 以及现有 `docs/` 资料。  
目的：记录当前项目中值得优化的目录层级、模块抽象、代码复用、测试性和潜在性能/维护风险。

> 本文只描述可优化内容和建议方向，不包含具体接口设计，也不要求立即改代码。真正实施前，应按项目要求先对目标符号做影响分析，再分阶段改动。

## 1. 当前架构概览

项目是一个 JXMIS Chrome 扩展，核心能力包括：

- 日报审核页：批量审批未审批日报。
- 周报填报页：批量报工、读取日报实际工时、生成 AI 周报总结、生成下周 WBS 计划并保存。
- 周报审核列表页：批量审核待审核周报。
- 扩展后台：保存 AI 配置、拉取模型列表、代理 OpenAI-compatible 流式总结请求。
- 扩展弹窗：配置模型地址、密钥、模型和系统提示词。

当前目录层级很扁平：

```text
.
├── background.js
├── content.js
├── defaults.js
├── manifest.json
├── page-batch-approve.js
├── page-batch-weekly-approve.js
├── page-batch-work.js
├── popup.css
├── popup.html
├── popup.js
└── docs/
```

这种结构对早期快速迭代很直接，但随着 `page-batch-work.js` 承载越来越多业务，代码的 locality 开始变差：理解一个行为需要在同一个大文件中跨越多个不相关段落；测试也缺少清晰的 module interface。

## 2. 优化优先级总览

| 优先级 | 候选优化 | 建议强度 | 主要收益 |
|---|---|---:|---|
| P0 | 深化批量报工 module | Strong | 提升核心工作流 locality，形成可测试 interface |
| P1 | 抽出 JXMIS 页面传输 adapter | Strong | 消除跨页面脚本重复的 fetch、post、错误处理 |
| P1 | 深化 content 自动化控制 module | Strong | 统一三套按钮/status/running 状态逻辑 |
| P2 | 合并周报详情读取与规范化重复逻辑 | Worth exploring | 避免同一周报 shape 在两个脚本中漂移 |
| P2 | 明确 AI 周报总结 seam | Speculative | 让流式总结、缓存、字段回填更容易测试 |
| P3 | 建立轻量测试和目录结构 | Worth exploring | 为后续重构提供保护网 |

## 3. 候选一：深化批量报工 module

涉及文件：

- `page-batch-work.js`
- 重点位置：`runBatchWork`、`buildNextExecutionRows`、日报实际工时匹配、AI 周报总结、下周 WBS 插入。

### 现状

`page-batch-work.js` 当前约 2500 多行，是项目中最重的文件。它同时承担：

- 页面上下文识别和基础请求。
- 周报上下文读取。
- 查询本周日报 taskDetail。
- 将日报实际工时、完成率、实际结束时间匹配到当前周执行表。
- 操作 DataTables change store。
- 生成 AI 周报总结。
- 读写周报总结缓存。
- 查询项目计划 WBS。
- 生成下周 WBS 执行计划。
- 操作原生新增行。
- 控制保存顺序和异常上报。

其中 `runBatchWork` 是核心编排函数，直接跨越多个 module seam。它的外部入口很小，但内部 implementation 同时知道页面表格、日报、周报、AI、WBS、保存顺序等细节。

### 主要问题

1. **locality 不稳定**

   修改“日报实际工时匹配”时，容易碰到“AI 周报总结”或“下周 WBS 计划”的上下文；修改“下周 WBS 计划”时，又需要理解当前周保存逻辑。复杂度没有集中在一个清晰的 module 内部，而是混在一条长编排路径中。

2. **interface 不是天然测试面**

   目前很多逻辑只能通过真实 JXMIS 页面、DataTables、`WkFormJS.saveAll()` 和浏览器消息来验证。`buildNextExecutionRows` 这类纯度较高的逻辑虽然更适合测试，但仍和日志、统计样本、WBS shape 处理耦合在一个大函数里。

3. **认知跳转成本高**

   从“本周日报实际工时”到“本周执行情况 AI 总结”再到“下周 WBS 计划生成”，属于不同业务概念，却放在同一个文件和同一条编排函数中。后续让 AI 或新人维护时，导航成本偏高。

### 建议方向

建议先不急着大规模拆目录，而是先围绕批量报工形成更深的 module：

- 保留“批量报工”作为外部可见的 page action。
- 将以下 implementation 收进内部子模块或同目录文件：
  - 周报上下文读取。
  - 日报实际数据获取与匹配。
  - 当前周执行表变更生成。
  - AI 周报总结生成。
  - 下周 WBS 候选筛选与执行计划生成。
  - 页面表格写入和保存编排。
- 让测试尽量从“输入周报上下文 + 日报行 + WBS 行，得到变更计划/插入计划”这个 interface 进入，而不是直接测 DOM 操作细节。

### 预期收益

- **locality**：日报匹配 bug、WBS 计划 bug、AI 总结 bug 分别集中到对应 implementation。
- **leverage**：一个批量报工 interface 可以覆盖多个内部行为，测试不必直接穿透所有小函数。
- **可读性**：`runBatchWork` 变成高层编排，具体规则落在命名清晰的 module 内。
- **回归保护**：可以先为 WBS 生成、日报匹配、保存决策补测试，再逐步移动代码。

### 风险与注意事项

- 这是当前项目中风险最高也最有价值的优化。实际改动前必须做影响分析。
- 不建议一次性把 2500 行拆成很多文件。优先围绕一条业务行为形成深 module，避免制造更多 shallow module。
- DOM/DataTables 操作应尽量作为 adapter 留在边缘，业务计算逻辑不要依赖真实页面。

## 4. 候选二：抽出 JXMIS 页面传输 adapter

涉及文件：

- `page-batch-approve.js`
- `page-batch-weekly-approve.js`
- `page-batch-work.js`

### 现状

三个页面脚本中重复出现了类似逻辑：

- `post(type, message, extra)`：向 content script 发状态消息。
- `sleep` / `randomDelay`：批量处理时的等待策略。
- `getWebapp` / `getBaseUrl`：拼接 JXMIS webapp 基础路径。
- `assertOk`：处理 HTTP 非 2xx。
- `fetchJson` 或重复的 same-origin fetch headers。

这些 implementation 现在分散在不同 page script 中。日报审核、周报审核、批量报工都在各自维护一份相似的 transport 行为。

### 主要问题

1. **重复 implementation 容易漂移**

   例如请求 header、`cache: "no-store"`、错误信息格式、baseUrl 拼接规则只要有一处改动，其他脚本很容易漏改。

2. **message seam stringly typed**

   页面脚本和 `content.js` 之间通过字符串 type/source 通信。现在每个脚本自己 post，消息协议没有集中定义，后续新增状态或请求容易出现拼写和处理分支不一致。

3. **删除测试说明它值得存在**

   如果删除一个共享 transport adapter，复杂度会重新散回三个页面脚本，说明这是一个真实 seam，而不是为了抽象而抽象。

### 建议方向

建立一个共享的 JXMIS 页面传输 adapter，用于承载：

- same-origin GET/POST JSON 请求。
- `assertOk` 统一错误处理。
- base URL / webapp 解析。
- 页面到 content 的消息发送。
- 批量操作 delay 策略。

需要注意：这不一定要立刻引入复杂构建工具。Chrome 扩展当前是直接加载脚本，可以先选择最小变更方式：

- 通过 `manifest.json` 的 `web_accessible_resources` 增加共享脚本。
- 或者先在单文件内部提取一组共享 helpers，验证形状后再移动。
- 或者后续引入构建步骤，但这属于更大架构决策，不建议作为第一步。

### 预期收益

- **locality**：请求/消息协议错误集中到一个 adapter。
- **leverage**：日报审核、批量报工、周报审核三个页面脚本复用同一套 transport 行为。
- **测试性**：可以替换成 fake adapter 测批量流程，不必真实请求 JXMIS。

### 风险与注意事项

- 当前脚本运行在页面上下文，不是普通 ES module 环境。拆分方式必须符合 Chrome 扩展加载机制。
- 不要把所有业务请求都塞进 transport adapter；adapter 只负责传输和协议，不负责日报/周报/WBS 业务规则。

## 5. 候选三：深化 content 自动化控制 module

涉及文件：

- `content.js`

### 现状

`content.js` 同时负责：

- 判断当前页面是哪类 JXMIS 页面。
- 注入对应页面脚本。
- 插入日报审核按钮和状态。
- 插入批量报工按钮和状态。
- 插入周报审核按钮和状态。
- 处理三类 page script 发回的状态消息。
- 转发 AI summary 流式消息。
- 转发周报总结缓存 get/set。
- 刷新页面或刷新 DataTables。

其中 `setDailyStatus`、`setWorkStatus`、`setWeeklyStatus` 形状高度相似：都是拿 status、button、running flag，设置文本、颜色、disabled、opacity、cursor 和按钮文案。

### 主要问题

1. **三套状态控制是 shallow module**

   每套实现都不深，interface 几乎等于 implementation：传入文字和 running，然后重复设置 DOM 状态。它们带来的 leverage 不高，维护时还容易漏改某一套。

2. **页面识别、UI 注入、消息桥混在一起**

   `content.js` 的 seam 不清晰。理解一个按钮的状态变化，需要同时看页面识别、按钮创建、消息处理、刷新策略。

3. **扩展新页面成本会上升**

   如果以后再加新的 JXMIS 页面自动化，很可能复制第四套 `ensureXxxPanel`、`setXxxStatus`、message handling 分支。

### 建议方向

把 content 自动化深化为“页面自动化注册 + 共享控制状态 + 消息桥”：

- 页面识别：每类页面提供自己的 matcher。
- 脚本注入：每类页面声明需要注入的 page script。
- 控件配置：每类页面声明按钮 id、status id、默认文案、running 文案、挂载位置。
- 共享控制状态 implementation：统一处理 disabled、文本、颜色、透明度、cursor。
- 消息桥：集中处理 source/type 到状态更新或 runtime message 的映射。

同样，不建议一开始设计复杂框架。可以先从 `setDailyStatus` / `setWorkStatus` / `setWeeklyStatus` 合并成一个小而深的 status control module 开始。

### 预期收益

- **locality**：按钮状态和 running 状态逻辑集中。
- **leverage**：三类页面复用一套 control implementation。
- **可扩展性**：新增页面时只增加配置和少量 adapter，不复制完整状态管理。
- **测试性**：可以用 DOM fake 验证状态控制，不需要真实 JXMIS 页面。

### 风险与注意事项

- 三类页面的按钮挂载位置不同，抽象时不要强行把挂载逻辑统一成复杂参数。
- 先合并真正重复的状态控制，再考虑页面注册结构。避免为了目录好看而制造 shallow module。

## 6. 候选四：合并周报详情读取与规范化重复逻辑

涉及文件：

- `page-batch-work.js`
- `page-batch-weekly-approve.js`

### 现状

两个脚本中都存在周报详情相关逻辑：

- `fetchWeeklyById`
- `normalizeWeeklyDetail`

批量报工需要读取当前周报上下文和周报详情；周报审核需要重新读取待审核周报，确认负责人和状态后再审核。两者访问的是相近的 JXMIS 周报数据 shape。

### 主要问题

1. **同一数据 shape 重复规范化**

   如果 JXMIS 返回字段变化，两个脚本都需要同步调整。

2. **业务语义可能不同**

   批量报工关注当前填报上下文，周报审核关注待审核状态和负责人校验。它们共享的是数据 shape，不一定共享完整业务规则。

3. **过早合并可能制造错误 seam**

   如果两个调用方只是偶然访问同一 endpoint，但 invariants 不同，强行合并会降低 locality。

### 建议方向

先做一轮轻量梳理：

- 对比两个 `normalizeWeeklyDetail` 的字段含义和默认值。
- 标注哪些字段是“周报详情通用 shape”，哪些字段是“批量报工专用”，哪些字段是“周报审核专用”。
- 只把真正通用的 shape normalization 放到共享 module。
- 业务判断仍留在各自 caller 内。

### 预期收益

- **locality**：周报详情字段适配集中。
- **leverage**：两个调用方复用同一份 shape normalization。
- **安全性**：保留不同业务规则，避免 fake depth。

### 风险与注意事项

- 这是 “Worth exploring”，不是马上强推的 Strong 优化。
- 先用 deletion test 判断：删除共享 normalization 是否会让字段适配复杂度散回两个调用方。如果会，说明值得抽。

## 7. 候选五：明确 AI 周报总结 seam

涉及文件：

- `page-batch-work.js`
- `content.js`
- `background.js`
- `popup.js`
- `defaults.js`

### 现状

AI 周报总结横跨多个运行环境：

1. `page-batch-work.js` 生成 prompt、请求总结、写入 `本周执行情况` 字段。
2. `content.js` 接收页面消息，创建 `chrome.runtime.connect` port。
3. `background.js` 读取配置，请求 OpenAI-compatible `/chat/completions`，解析 SSE 流。
4. `content.js` 把 chunk/done/error 转回页面脚本。
5. `page-batch-work.js` 处理 chunk 并回填字段。
6. `popup.js` 负责配置 baseUrl、apiKey、model、systemPrompt。

这是一个真实 seam，因为 Chrome 扩展限制决定了页面上下文、content script、background worker 必须通过 adapter 通信。

### 主要问题

1. **生命周期分散**

   requestId、chunk、done、error、disconnect、cache get/set 分散在三个文件中。排查流式总结失败时，需要跨文件追踪。

2. **测试困难**

   没有一个单独 interface 能表达“给定 prompt 和 fake stream，最终字段如何变化”。现在更接近端到端手动验证。

3. **错误处理路径多**

   模型未配置、模型请求失败、SSE 解析失败、port disconnect、缓存读写失败、页面字段不存在，都分散在不同位置处理。

### 建议方向

此项建议强度是 Speculative。只有当 AI 周报总结继续频繁变化时，才值得深化。

可以考虑形成一个 AI weekly summary module，内部包含：

- prompt 构造。
- cache key 和 cache payload。
- stream lifecycle。
- chunk 聚合和字段回填。
- 错误分类。

页面、content、background 仍然作为 adapter 存在，因为运行环境不能合并。

### 预期收益

- **locality**：AI 总结相关错误集中。
- **leverage**：可以通过 fake stream adapter 测试总结生命周期。
- **测试性**：将“流式输入 -> 字段输出/错误状态”变成稳定测试面。

### 风险与注意事项

- 当前 seam 是 Chrome 扩展天然要求，不要为了抽象把运行环境差异隐藏到难懂的间接层里。
- 如果 AI 总结功能近期稳定，不建议优先做。

## 8. 目录层级优化建议

当前根目录所有运行时代码平铺，短期可接受，但随着逻辑增长，建议逐步演进为按运行环境和业务 module 分组。

一个可能方向：

```text
src/
├── background/
│   └── ai-summary-runtime.js
├── content/
│   ├── automation.js
│   ├── controls.js
│   └── message-bridge.js
├── page/
│   ├── daily-approval/
│   ├── batch-work/
│   ├── weekly-approval/
│   └── shared/
├── popup/
└── shared/
```

但这不是第一步。当前项目没有构建流程，直接上 `src/` 可能引入额外复杂度。更稳妥的顺序是：

1. 先在现有文件内收敛重复 implementation。
2. 为高价值纯逻辑补测试。
3. 再按运行环境移动文件。
4. 最后再考虑是否引入打包或模块化加载。

## 9. 测试与验证建议

当前项目主要依赖真实页面手动验证，建议逐步补以下测试面：

### 9.1 纯逻辑测试

优先覆盖：

- 下周 WBS 计划生成：
  - 日期范围相交。
  - 工作日/节假日。
  - `待定` owner。
  - 无 owner 且无 duration。
  - 24 小时拆分。
  - 已存在行去重。
- 日报实际工时匹配：
  - 按 WBS 匹配。
  - 按人员/任务名匹配。
  - 多条日报聚合。
  - fallback 到计划值。
- 周报详情 normalization：
  - 缺字段。
  - 字段类型变化。
  - 状态/负责人判断。

### 9.2 Adapter 测试

优先覆盖：

- JXMIS page transport adapter：
  - baseUrl 拼接。
  - HTTP 非 2xx 错误。
  - JSON 解析。
  - message payload shape。
- content 控制状态：
  - running true/false 时按钮文案和 disabled。
  - status 文案和颜色。
  - 页面元素不存在时不抛错。

### 9.3 手动回归清单

每次重构后至少手动验证：

- 日报批量审批能启动、进度更新、完成后刷新。
- 批量报工能写当前周执行表、生成周报总结、生成下周计划并保存。
- 周报批量审核能预览、确认、逐条审核、完成后刷新。
- popup 能保存配置、刷新模型列表。
- AI summary 流式输出、done、error 都能正确反馈到页面。

## 10. 建议实施顺序

### 第一阶段：低风险复用

- 合并 `content.js` 中三套 status control implementation。
- 收敛 `post` / `sleep` / `randomDelay` / `assertOk` 等重复 helper。
- 不移动大文件，不改核心业务顺序。

### 第二阶段：批量报工核心规则测试化

- 为 `buildNextExecutionRows` 附近的 WBS 规则建立测试。
- 为日报实际工时匹配建立测试。
- 将 DOM/DataTables 写入作为 adapter 留在边缘。

### 第三阶段：深化批量报工 module

- 拆分批量报工内部 implementation。
- 保持外部 page action interface 不变。
- 每次只移动一个概念：日报匹配、AI 总结、WBS 计划、保存编排。

### 第四阶段：目录结构演进

- 在测试保护下把 page/content/background/popup 按运行环境分组。
- 如果需要，再评估是否引入构建流程。

## 11. 结论

当前项目最值得优化的不是“把文件拆小”本身，而是让核心业务形成更深的 module：

- 批量报工应当成为一个外部 interface 小、内部 implementation 深的 module。
- 页面传输和 content 控制状态是明确的复用点，适合先做低风险优化。
- 周报详情和 AI summary seam 需要先验证业务 invariants，再决定是否深化。

优先建议：先处理 `page-batch-work.js` 的批量报工 module，但实施前先用更小的复用和测试任务铺路，避免一次性重构带来回归风险。
