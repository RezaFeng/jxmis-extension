# 经营分析 UAT 与双跑验收

本文用于真实 JXPMO 的只读验收、旧工具双跑对账和迁移签字。离线 fixture 通过不等于真实业务验收通过。

## 1. 当前状态

| 项目 | 状态 | 证据/说明 |
|---|---|---|
| 生产构建 | 已通过 | 2026-07-18 `npm run verify` |
| Node 单元/集成测试 | 已通过 | 188/188 |
| 离线 Chromium E2E | 已通过 | 9/9，且无真实 JXPMO 请求 |
| 真实系统只读 UAT | 待执行 | 当前工作区未提供可复用的已登录浏览器会话 |
| 3 部门 × 2 周双跑 | 待执行 | 需要部门选择、旧工具基线和业务负责人参与 |
| 产品负责人签字 | 待执行 | 对账和预期差异确认后完成 |
| 旧工具归档 | 禁止执行 | 上述门禁尚未满足 |

当前实现和文档准备已完成，但不能在没有有效登录会话、真实部门选择和签字人的情况下伪造双跑结果。

## 2. 验收角色

| 角色 | 职责 |
|---|---|
| 执行人 | 在已登录 JXPMO 中完成只读查询、截图和导出 |
| 数据复核人 | 比较项目集合、来源完整性和逐指标结果 |
| 产品负责人 | 确认预期修正差异并签字决定切换/归档 |
| 扩展维护人 | 记录版本、修复缺陷并保留自动化验证结果 |

至少两人完成验收：执行/复核可由同一人承担，产品负责人必须独立确认最终结论。

## 3. 前置条件

- 使用测试或具备只读权限的真实账号登录 `https://jxmis.cyberwing.cn/jxpmo/`。
- 安装本分支构建的 `dist/`，记录 commit hash、Chrome 版本和扩展版本。
- 在 Options 中确认经营筛选和预警阈值；不要录入或截图 API Key。
- 选择至少 3 个有效部门，每个部门在两个完整自然周均有可核对项目。
- 选取连续两个已结束自然周，建议避开当前进行中周。
- 准备旧工具的只读副本和对应版本。用户提供路径带尾随 `#`，本机实际可读目录为 `/mnt/d/Downloads/devtools/weekly_report_toolkit_V3.1/weekly_report_toolkit_V3.1/`。
- 禁止把旧工具 `config.js`、Cookie、JSESSIONID、项目映射或 `output/` 原始数据复制到扩展仓库。

## 5. 真实系统只读场景

除原有审批/报工功能的既有 UAT 外，经营分析至少执行以下场景。查询和导出允许；不得向 JXPMO 写入经营数据。

| ID | 场景 | 操作 | 期望 |
|---|---|---|---|
| BA-01 | 菜单进入/离开 | 在项目模块点击“经营分析”，再点击其他项目菜单 | 只替换右侧主内容；离开后原页面和菜单状态恢复；无重复菜单 |
| BA-02 | 默认日期 | 首次进入报告 | 日期为最近完整周一至周日 |
| BA-03 | 部门范围 | 切换 3 个部门并查询 | 下拉显示配置筛选后的项目数；项目均按部门 ID 归属 |
| BA-04 | 项目分页 | 选择项目数较多部门 | 项目数与系统全分页结果一致，不在 200/2000 条处截断 |
| BA-05 | 完整报告 | 查询完整自然周 | 35 项值、全部区块、来源覆盖率和正式状态可见 |
| BA-06 | 自定义区间 | 查询跨周或跨月日期 | 文案使用本期/上期/下期；结束日决定月与逾期截止 |
| BA-07 | 临时选择 | 搜索、筛选、勾选项目并恢复 | 临时卡片联动；恢复全量可用；导出仍为正式口径 |
| BA-08 | 实时与刷新 | 重复查询和刷新同一周期 | 每次都发起实时请求；查询时间更新；相邻上期由筛选周期推导，不读取快照 |
| BA-09 | 全部部门 | 直接选择全部部门查询 | 一次实时采集覆盖全部有效部门；公共来源不按部门重复；项目按 ID 去重后重算比率 |
| BA-10 | 部分失败 | 在可控测试条件下制造单项目来源失败 | 可用部分保留，技术失败显示“未获取”，仅重试失败项且上下文只在页面内存 |
| BA-11 | 会话失效 | 在测试账号会话过期后查询 | 停止后续请求，显示“登录已失效”；重新登录后可重试 |
| BA-12 | HTML 导出 | 导出部门和全部部门报告，断网打开 | 页面可展开、排序、筛选、恢复；无外部请求；文件名含部门和日期 |
| BA-13 | 本地存储 | 升级旧版后查看 Options、IndexedDB 和多次查询 | 旧 `cw-business-analytics` 被幂等删除；不再创建；Options 无经营缓存/历史清理按钮 |
| BA-14 | 安全检查 | 查看 DevTools network/storage 和导出源码 | 无经营数据发往 AI；无快照；无 Cookie/JSESSIONID/API Key/人员级日报进入导出 |
| BA-15 | 回款合同与红冲 | 查询健新科技 2026-07 报告，反查订单合同和 2025 红冲计划 | `contractNum` 可命中逗号分隔的项目 `contractNo`；跨项目部门仍按合同关联；同一 `planId` 的正负金额净额为 0 时不显示逾期 |
| BA-16 | 回款失败与异常 | 制造回款接口失败，并准备非法 `recFlag`/金额 fixture | 失败时卡片显示“未获取”且不调用旧接口；异常行进入诊断但不污染金额、笔数和回款率 |

## 9. 预期修正差异

以下差异不是自动判定为通过；复核人必须确认它确实由新规则导致，而非实现缺陷。

| 主题 | 旧工具口径 | 扩展口径 | 验收证据 |
|---|---|---|---|
| 项目范围 | 手工大区项目编码白名单 | 组织树 + 项目主数据 + 本地筛选 + 部门 ID | 项目集合差异清单 |
| 登录 | `config.js` 中手工凭证 | 当前页面同源会话，不持久化 JSESSIONID | storage/导出检查 |
| 软件服务合同 | 基线/增强文件可能覆盖 | `ProjectInfoService.subcontractAmount` | 项目字段抽样 |
| 投入成本 | 旧版曾缺失或使用固定单价 | 日报 `cost` 唯一来源 | 日报聚合抽样 |
| WBS 截止 | 未来完成记录可能穿透 | `actualEndTime <= endDate` | 历史周期 WBS 抽样 |
| 周报选择 | 列表首条或单周规则 | 按日期交集，周期内取最新有效版，跨周聚合 | 周报 ID/周期清单 |
| 回款归属 | 旧双接口字段不一致，整串合同号等值匹配并过滤负数红冲 | `queryInvoicePlanDetailService` 单一来源；销售部门控制查询范围；全量项目合同拆分后精确关联；按 `planId` 保留带符号金额净额 | 接口请求、合同候选、未映射/多匹配/异常诊断和红冲净额抽样 |
| 空值与失败 | 增强失败时可能显示 0/空 | 成功返回的空业务值和零分母为 0；技术失败为“未获取” | 空字段与人为失败场景 |
| 正式/临时口径 | 页面筛选可能直接改变输出 | 临时选择不覆盖顶部正式统计和导出 | 导出前后对比 |
| 会议待办 | 旧报告包含 | 不在经营分析范围 | 产品负责人确认 |

## 11. 签字与切换门禁

以下条件必须全部满足：

- 6 份部门样本的项目集合已解释并确认。
- 35 项核心值完成逐类对账，所有差异归类为一致、预期修正或已修复缺陷。
- 部分失败、会话过期、大项目集、跨周/跨月和离线导出已验证。
- 自动化 `npm run verify` 在最终 commit 上通过。
- 导出和本地存储未发现 Cookie、JSESSIONID、API Key 或人员级日报泄露。
- 无阻断/高严重度缺陷；中低问题有明确处置决定。

## 12. 旧工具归档

产品负责人签字“同意切换”后才能归档：

1. 记录旧工具仓库路径、版本/文件校验值和最后一次对账日期。
2. 删除或隔离旧副本中的真实 Cookie、JSESSIONID、API Key 和临时导出；不要把凭证纳入归档。
3. 将旧工具标记为只读/Deprecated，README 指向本扩展和本 UAT 证据。
4. 保留必要的对账表、差异说明和产品签字，不保留不必要的人员级原始数据。
5. 确认生产出报只使用扩展，旧 Node/Python/Playwright/Excel 流水线不再执行。

若任一门禁未满足，旧工具保持可读基线状态，不执行删除、迁移或归档操作。
