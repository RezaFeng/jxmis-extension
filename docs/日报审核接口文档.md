# 日报审核接口文档

## 1. 文档范围

本文档基于当前目录内两类抓包导出内容整理：

- `日报审核curl脚本.bat`
- `jxmis.cyberwing.cn/...` 下的 raw response 文件

本次抓包覆盖的是 **“日报审批”页面** 的访问与审批流程，不是员工侧“日报填报新建/保存”流程。

因此本文档中“日报提交”指的是：

- 经理/审核人对日报执行“同意/不同意”的审批提交

而不是：

- 员工自己新建日报并提交日报

## 3. 总体调用链路

### 3.1 实际抓包链路

1. 访问主框架页
   - `GET /jxpmo/index/frame`
2. 获取当前登录用户
   - `GET /jxpmo/rest/org/user`
3. 获取菜单
   - `GET /jxpmo/rest/org/ManagerService/getMenu?format=json&allData=true`
4. 打开“日报审批”页面
   - `GET /jxpmo/project/ProjectRapportService/dailyApprovalPage`
5. 加载未审批日报列表
   - `GET /jxpmo/rest/project/queryDailyApprovalService/query?...approval_state=0&projectManager={userId}...`
6. 加载已审批日报列表
   - `GET /jxpmo/rest/project/queryDailyApprovalService/query?...approval_state=1&projectManager={userId}...`
7. 执行审批提交
   - `POST /jxpmo/rest/project/ProjectRapportService/batchDailyApproval`
8. 再次刷新未审批列表
   - `GET /jxpmo/rest/project/queryDailyApprovalService/query?...approval_state=0&projectManager={userId}...`

### 3.2 页面内的关键关系

`dailyApprovalPage` 页面中直接写死了两个 DataTable 数据源：

- 未审批日报：
  - `/rest/project/queryDailyApprovalService/query?queryName=queryList&queryType=page&approval_state=0&projectManager=e7c45b074bd54616afadc1df9fbe5a0c`
- 已审批日报：
  - `/rest/project/queryDailyApprovalService/query?queryName=queryList&queryType=page&approval_state=1&projectManager=e7c45b074bd54616afadc1df9fbe5a0c`

因此，**日报列表查询依赖当前登录人的 `userId` 作为 `projectManager` 条件**。

## 6. 页面入口接口

### 6.1 打开日报审批页面

**接口**

```http
GET /jxpmo/project/ProjectRapportService/dailyApprovalPage
```

**用途**

- 返回日报审批页面 HTML
- 页面内包含两张表：未审批日报、已审批日报
- 页面内直接声明列表查询 URL

**页面中确认到的关键前端逻辑**

1. 未审批页签使用 `approval_state=0`
2. 已审批页签使用 `approval_state=1`
3. 搜索框通过 `likeAll` 追加搜索词
4. 单条审批操作按钮调用：

```javascript
DailyApproval.singleDailyApproval(
  '1' or '2',
  row.id,
  row.type,
  row.createTime,
  row.extId,
  row.realFinishRate,
  row.planTime,
  encodeURIComponent(row.taskName || '')
)
```

其中：

- `'1'` 表示同意
- `'2'` 表示不同意

这是从页面按钮 onclick 直接确认到的。

## 7. 日报列表查询接口

### 7.1 接口定义

**接口**

```http
GET /jxpmo/rest/project/queryDailyApprovalService/query
```

**用途**

- 获取日报审批列表
- 同一个接口通过 `approval_state` 区分未审批和已审批

### 7.2 已确认的请求参数

抓包中出现过的完整请求示例：

```http
GET /jxpmo/rest/project/queryDailyApprovalService/query?queryName=queryList&queryType=page&approval_state=0&projectManager=e7c45b074bd54616afadc1df9fbe5a0c&draw=1&page=1&start=0&length=50&rows=50
```

以及：

```http
GET /jxpmo/rest/project/queryDailyApprovalService/query?queryName=queryList&queryType=page&approval_state=1&projectManager=e7c45b074bd54616afadc1df9fbe5a0c&draw=1&page=1&start=0&length=50&rows=50
```

### 7.2.1 请求体结论

此接口当前可判断为：

- `GET` 接口
- **无请求体**
- 参数全部放 URL query string

依据：

1. `日报审核curl脚本.bat` 里此接口请求无 `--data` / `--data-raw`
2. 页面 `dailyApprovalPage` 里表格 `data-url` 直接写完整查询参数
3. 搜索逻辑也是把 `likeAll` 直接拼到 URL 后面，再触发 DataTables reload

因此，此接口应按下面方式理解：

- 有“请求参数结构”
- 没有“POST/PUT 那种 JSON 请求体结构”

### 7.3 参数说明

| 参数名 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| `queryName` | string | 是 | 固定为 `queryList` |
| `queryType` | string | 是 | 固定为 `page` |
| `approval_state` | string/int | 是 | 审批状态：`0` 未审批，`1` 已审批 |
| `projectManager` | string | 是 | 项目经理/审批人用户 ID，来自 `/rest/org/user` |
| `draw` | int | 否 | DataTables 绘制计数器 |
| `page` | int | 否 | 页码，抓包中从 `1` 开始 |
| `start` | int | 否 | 分页起始偏移量 |
| `length` | int | 否 | 每页条数 |
| `rows` | int | 否 | 每页条数，抓包中与 `length` 一致 |
| `likeAll` | string | 否 | 页面搜索框追加的模糊搜索条件 |

### 7.4 搜索机制

页面 HTML 里搜索框 `data-url` 为：

```text
/rest/project/queryDailyApprovalService/query?queryName=queryList&queryType=page&approval_state=0&projectManager=...&likeAll=
```

说明页面搜索是：

- 把用户输入值做 `encodeURIComponent`
- 直接拼到 `likeAll=` 后面

即请求形态应为：

```http
GET /jxpmo/rest/project/queryDailyApprovalService/query?queryName=queryList&queryType=page&approval_state=0&projectManager={userId}&likeAll={keyword}
```

## 8. 日报审批提交接口

### 8.1 接口定义

**接口**

```http
POST /jxpmo/rest/project/ProjectRapportService/batchDailyApproval
Content-Type: application/json
X-Requested-With: XMLHttpRequest
```

**用途**

- 提交日报审批结果
- 页面按钮包括：
  - 单条同意
  - 单条不同意
  - 批量同意
  - 批量不同意

### 8.2 已确认事实

本次抓包里明确确认到：

1. 请求方法为 `POST`
2. `Content-Type` 为 `application/json`
3. 请求目标为 `/rest/project/ProjectRapportService/batchDailyApproval`
4. 请求体是 **JSON 数组**，单条审批时数组内也是 1 个对象
5. 请求体字段已确认包含 `id/type/state/createTime/realFinishRate/planTime/extId/approvalTimely/achievementComplete/achievementQuality/approvalComment`
6. 响应体为：

```json
"success"
```

7. 响应后再次拉取未审批列表时，数据量减少，说明审批动作生效

### 8.3 从页面脚本确认到的审批参数来源

页面中单条审批按钮调用：

```javascript
DailyApproval.singleDailyApproval(
  approvalState,
  id,
  type,
  createTime,
  extId,
  realFinishRate,
  planTime,
  taskName
)
```

因此，**单条审批前端至少掌握以下字段**：

- `approvalState`
- `id`
- `type`
- `createTime`
- `extId`
- `realFinishRate`
- `planTime`
- `taskName`

其中：

- `taskName` 是前端弹窗/上下文参数
- 它 **未出现在你补充的已确认 POST body 中**
- 因此组装接口请求时，应以 8.5 节的 body 字段为准

### 8.4 审批状态枚举

页面按钮已明确：

| 值 | 含义 |
|---|---|
| `1` | 同意 |
| `2` | 不同意 |

### 8.5 请求体说明

#### 8.5.1 已确认请求体示例

```json
[
  {
    "id": "42fabb1a604a4f35bca79bb3ccf3b468",
    "type": "task",
    "state": "1",
    "createTime": "2026-04-20 17:30:00",
    "realFinishRate": "33",
    "planTime": "24",
    "extId": "d743368349b34991b09eae525fb7cc47",
    "approvalTimely": "1",
    "achievementComplete": "1",
    "achievementQuality": "1",
    "approvalComment": ""
  }
]
```

#### 8.5.2 字段说明

| 字段 | 类型 | 是否必需 | 说明 |
|---|---|---:|---|
| `id` | string | 是 | 日报主键，对应列表返回的 `id` |
| `type` | string | 是 | 日报来源类型，抓包中为 `task` |
| `state` | string | 是 | 审批结果：`1` 同意，`2` 不同意 |
| `createTime` | string | 是 | 日报填报时间 |
| `realFinishRate` | string | 是 | 审批确认进度，抓包中以字符串传输，如 `"33"` |
| `planTime` | string | 是 | 计划工时，抓包中以字符串传输，如 `"24"` |
| `extId` | string | 是 | 关联业务主键，对应列表返回的 `extId` |
| `approvalTimely` | string | 是 | 审批及时性评分，抓包值为 `"1"` |
| `achievementComplete` | string | 是 | 成果完整性评分，抓包值为 `"1"` |
| `achievementQuality` | string | 是 | 成果质量评分，抓包值为 `"1"` |
| `approvalComment` | string | 是 | 审批意见，允许空字符串 |

#### 8.5.3 字段关系说明

需要注意两点：

1. 页面按钮里传给前端函数的审批结果字段语义是 `approvalState`
2. 实际 POST body 里对应字段名是 `state`

也就是说，在调用接口时应以 **`state`** 为准，而不是 `approvalState`。

#### 8.5.4 批量提交规则

接口名为 `batchDailyApproval`，且请求体最外层是数组，因此批量审批时应直接传多个对象，例如：

```json
[
  { "...": "..." },
  { "...": "..." }
]
```

每个对象结构与单条审批完全一致。

### 8.6 响应说明

抓包中接口响应内容为：

```json
"success"
```

说明该接口不是常见的对象结构响应，而是直接返回字符串。

## 12. 最终结论

### 12.1 如何获取日报列表

核心步骤是：

1. 通过 `JSESSIONID` 保持登录态
2. 调用 `/rest/org/user` 获取当前登录用户 `userId`
3. 调用 `/rest/project/queryDailyApprovalService/query`
4. 把 `projectManager` 设置为当前 `userId`
5. 用 `approval_state=0/1` 区分未审批/已审批
6. 用 `likeAll` 做搜索

### 12.2 如何提交日报审批

核心步骤是：

1. 从列表接口拿到待审批日报行数据
2. 至少收集该行的 `id / extId / type / createTime / realFinishRate / planTime`
3. 设置 `state`
   - `1` 同意
   - `2` 不同意
4. 同时提交评分字段 `approvalTimely / achievementComplete / achievementQuality` 以及 `approvalComment`
5. `POST /rest/project/ProjectRapportService/batchDailyApproval`
6. 期望响应为 `"success"`
7. 再次查询未审批列表确认结果
