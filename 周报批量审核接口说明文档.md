# 周报批量审核接口说明文档

## 1. 文档概述

### 1.1 文档目的

本文档基于 `周报审批.saz` 抓包结果整理，用于开发实现“批量审核周报”功能。

核心目标：

- 获取当前登录用户。
- 查询当前用户作为生产负责人的周报。
- 筛选待审核周报。
- 批量提交批复意见。
- 校验批复结果。

### 1.2 抓包结论

周报审核链路不是一个完整的服务端批量接口，而是前端通过多个单条接口组合完成：

```text
获取当前用户
-> 查询周报列表
-> 前端筛选当前用户负责且待审核的周报
-> 对每条周报调用 addReply 批复接口
-> 查询结果确认状态变更
```

抓包中未发现专门的“批量批复”接口。批量功能应在新功能中通过循环调用单条批复接口实现，或由后端新增真正的批量接口封装该逻辑。

## 2. 业务状态与关键字段

### 2.1 周报状态

| 状态值 | 状态描述 | 说明 |
|---|---|---|
| `10` | 草稿 | 未提交，不能批复 |
| `20` | 待审核 | 可批复 |
| `30` | 已批复 | 已完成批复 |

实现时应优先使用 `status` 判断，不建议依赖 `statusDesc`，因为抓包文档里部分响应存在编码展示问题。

### 2.2 当前用户字段

当前用户接口返回中，批量筛选主要使用：

| 字段 | 说明 |
|---|---|
| `userId` | 当前用户在周报系统中的用户 ID |
| `userFullName` | 当前用户姓名，用于列表查询中的 `likeAll` 搜索 |
| `username` | 登录账号，通常不用于周报负责人匹配 |

### 2.3 周报列表关键字段

| 字段 | 说明 |
|---|---|
| `wkId` | 周报主键，批复接口必传 |
| `wkName` | 周报名称 |
| `wkNum` | 周报编号 |
| `prodPerson` | 生产负责人用户 ID |
| `prodPersonName` | 生产负责人姓名 |
| `status` | 周报状态 |
| `statusDesc` | 状态描述 |
| `replySummary` | 批复意见摘要 |
| `approvalTime` | 批复时间 |
| `year` | 年份 |
| `month` | 月份 |
| `weekDate` | 周期范围 |

批量审核候选条件：

```text
row.prodPerson == currentUser.userId
AND row.status == "20"
```

## 3. 接口清单

### 3.1 获取当前用户

#### 基本信息

| 项 | 内容 |
|---|---|
| 接口用途 | 获取当前登录用户信息 |
| 请求方式 | `GET` |
| 接口路径 | `/jxpmo/rest/org/user` |
| 是否鉴权 | 是 |
| 鉴权方式 | 依赖浏览器会话 Cookie，主要是 `JSESSIONID` |

#### 请求示例

```http
GET https://jxmis.cyberwing.cn/jxpmo/rest/org/user
Cookie: JSESSIONID=<当前登录会话>
X-Requested-With: XMLHttpRequest
Accept: application/json, text/javascript, */*; q=0.01
```

#### 响应示例

```json
{
  "userId": "e7c45b074bd54616afadc1df9fbe5a0c",
  "userFullName": "冯俊华",
  "username": "fengjunhua",
  "user": {
    "userId": "e7c45b074bd54616afadc1df9fbe5a0c",
    "userFullName": "冯俊华",
    "username": "fengjunhua"
  }
}
```

#### 开发要点

- 以后续批量筛选为准，取顶层 `userId` 和 `userFullName` 即可。
- 若顶层字段为空，可兼容读取 `user.userId` 和 `user.userFullName`。
- 当前接口本身不会返回待审周报，只提供当前用户身份。

### 3.2 查询周报列表

#### 基本信息

| 项 | 内容 |
|---|---|
| 接口用途 | 查询周报分页列表 |
| 请求方式 | `GET` |
| 接口路径 | `/jxpmo/rest/project/WkReportService/query` |
| 是否鉴权 | 是 |
| 返回类型 | 分页 JSON |

#### 请求参数

| 参数 | 是否必填 | 示例 | 说明 |
|---|---|---|---|
| `queryName` | 是 | `queryList` | 固定值，列表查询名称 |
| `filterQuery` | 是 | `true` | 固定值，启用过滤查询 |
| `queryType` | 是 | `page` | 固定值，分页查询 |
| `year` | 是 | `2026` | 查询年份 |
| `month` | 是 | `5` | 起始月份 |
| `monthEnd` | 否 | `5` | 结束月份，抓包中的搜索请求会带该字段 |
| `likeAll` | 否 | `冯俊华` | 全局搜索关键字，抓包中用当前用户姓名搜索 |
| `draw` | 是 | `1` | DataTables 请求序号 |
| `page` | 是 | `1` | 页码，从 1 开始 |
| `start` | 是 | `0` | 起始行偏移 |
| `length` | 是 | `25` | 每页条数 |
| `rows` | 是 | `25` | 每页条数，通常与 `length` 一致 |

#### 抓包请求示例

按月份查询：

```http
GET https://jxmis.cyberwing.cn/jxpmo/rest/project/WkReportService/query?queryName=queryList&filterQuery=true&queryType=page&year=2026&month=5&rows=25&draw=1&page=1&start=0&length=25
```

按当前用户姓名搜索：

```http
GET https://jxmis.cyberwing.cn/jxpmo/rest/project/WkReportService/query?queryName=queryList&filterQuery=true&queryType=page&year=2026&month=5&monthEnd=5&likeAll=%E5%86%AF%E4%BF%8A%E5%8D%8E&draw=1&page=1&start=0&length=25&rows=25
```

#### 响应结构

```json
{
  "pageNo": 1,
  "pageCount": 1,
  "pageSize": 25,
  "total": 4,
  "recordsTotal": 4,
  "recordsFiltered": 4,
  "rows": [
    {
      "wkId": "f55a2a2d01284285afe157f2dc454d2f",
      "wkName": "(2026)年(5)月第(1)周...项目周报",
      "wkNum": "JX-AJ2023031PWR20260501",
      "year": 2026,
      "month": 5,
      "weekDate": "2026-04-26 ~ 2026-05-02",
      "prodPerson": "e7c45b074bd54616afadc1df9fbe5a0c",
      "prodPersonName": "冯俊华",
      "status": "20",
      "statusDesc": "待审核",
      "replySummary": null,
      "approvalTime": null
    }
  ]
}
```

#### 开发要点

- `likeAll=当前用户姓名` 只是缩小搜索范围，不能替代权限和状态判断。
- 搜索结果仍可能包含草稿周报，必须继续过滤 `status == "20"`。
- 搜索结果可能包含当前用户姓名出现在其他字段中的周报，必须继续过滤 `prodPerson == 当前用户 userId`。
- 批量审核应遍历分页，不能只处理第一页。

### 3.3 查询单条周报审批状态

#### 基本信息

| 项 | 内容 |
|---|---|
| 接口用途 | 按 `wkId` 查询周报主信息，可用于批复前后校验 |
| 请求方式 | `GET` |

|---|---|---|---|
| `queryType` | 是 | `all` | 固定值 |
| `queryName` | 是 | `queryByProjectInfo` | 固定值 |
| `wkId` | 是 | `f55a2a2d01284285afe157f2dc454d2f` | 周报主键 |

#### 请求示例

```http
GET https://jxmis.cyberwing.cn/jxpmo/rest/project/queryByProjectInfosService/query?queryType=all&queryName=queryByProjectInfo&wkId=f55a2a2d01284285afe157f2dc454d2f
```

#### 批复前响应关键字段

```json
{
  "wkId": "f55a2a2d01284285afe157f2dc454d2f",
  "prodPerson": "e7c45b074bd54616afadc1df9fbe5a0c",
  "prodPersonName": "冯俊华",
  "status": "20",
  "replySummary": null,
  "approvalTime": null
}
```

#### 批复后响应关键字段

```json
{
  "wkId": "f55a2a2d01284285afe157f2dc454d2f",
  "prodPerson": "e7c45b074bd54616afadc1df9fbe5a0c",
  "prodPersonName": "冯俊华",
  "status": "30",
  "replySummary": "已核实",
  "approvalTime": "2026-05-01 20:59:28"
}
```

#### 开发要点

- 批量审核前可选调用，用于防止列表状态过期。
- 批量审核后建议调用，用于确认状态已变为 `30`。
- 如果批复接口返回成功但该接口仍返回 `20`，应记录为异常，避免重复提交。

| 项 | 内容 |
|---|---|
| 接口用途 | 获取批复意见输入表单 |
| 请求方式 | `GET` |
| 接口路径 | `/jxpmo/view/project/ScaleMarksService/MonthReportReplyPage` |
| 是否鉴权 | 是 |
| 返回类型 | HTML |

#### 请求示例

```http
GET https://jxmis.cyberwing.cn/jxpmo/view/project/ScaleMarksService/MonthReportReplyPage?_=1777641267129
```

#### 响应关键内容


#### 开发要点

- 该接口主要服务前端弹窗展示。
- 后端或脚本实现批量审核时通常不需要调用该接口。
- 真正提交批复的是 `addReply` 接口。

### 3.5 提交周报批复

#### 基本信息

| 项 | 内容 |
|---|---|
| 接口用途 | 对单条周报提交批复意见 |
| 请求方式 | `GET` |
| 接口路径 | `/jxpmo/rest/project/WkReportService/addReply` |
| 是否鉴权 | 是 |
| 返回类型 | JSON 字符串 |

#### 请求参数

| 参数 | 是否必填 | 示例 | 说明 |
|---|---|---|---|
| `format` | 是 | `json` | 固定值 |
| `wkId` | 是 | `f55a2a2d01284285afe157f2dc454d2f` | 周报主键 |
| `reply` | 是 | `已核实` | 批复意见，需要 URL 编码 |

#### 请求示例

```http
GET https://jxmis.cyberwing.cn/jxpmo/rest/project/WkReportService/addReply?format=json&wkId=f55a2a2d01284285afe157f2dc454d2f&reply=%E5%B7%B2%E6%A0%B8%E5%AE%9E
```

#### 响应示例

```json
"批复完成"
```

#### 开发要点

- 该接口是单条批复接口，批量审核需要循环调用。
- `reply` 参数放在 URL Query 中，必须做 URL 编码。
- 因为是 GET 修改数据，调用方要避免浏览器、代理或重试机制误触发重复请求。
- 建议批量调用前再次确认 `status == "20"`，避免对已批复或草稿数据提交。
- 调用成功后，服务端会把状态从 `20` 变更为 `30`，并写入 `replySummary`、`approvalTime`。

## 4. 批量审核推荐流程

### 4.1 输入参数

批量审核功能建议提供以下输入：

| 参数 | 说明 | 示例 |
|---|---|---|
| `year` | 周报年份 | `2026` |
| `month` | 周报月份 | `5` |
| `reply` | 统一批复意见 | `已核实` |
| `dryRun` | 是否只预览不提交 | `true / false` |

### 4.2 处理步骤

```text
1. 调用 /rest/org/user 获取当前用户。
2. 读取 currentUser.userId 和 currentUser.userFullName。
3. 分页调用 /WkReportService/query。
4. 查询参数带 year、month、monthEnd、likeAll=currentUser.userFullName。
5. 遍历 rows。
6. 筛选 prodPerson == currentUser.userId 且 status == "20" 的周报。
7. 展示或记录待批复清单。
8. 如果 dryRun=true，到此结束。
9. 对每条候选周报调用 /WkReportService/addReply。
10. 每条提交后调用 queryByProjectInfo 校验 status 是否变成 "30"。
11. 汇总成功、跳过、失败结果。
```

### 4.3 伪代码

```javascript
async function batchApproveWeeklyReports({ year, month, reply, dryRun = false }) {
  const currentUser = await getCurrentUser();
  const userId = currentUser.userId || currentUser.user?.userId;
  const userFullName = currentUser.userFullName || currentUser.user?.userFullName;

  const candidates = [];
  let page = 1;
  const pageSize = 25;

  while (true) {
    const result = await queryWkReports({
      year,
      month,
      monthEnd: month,
      likeAll: userFullName,
      page,
      start: (page - 1) * pageSize,
      length: pageSize,
      rows: pageSize
    });

    for (const row of result.rows || []) {
      if (row.prodPerson === userId && String(row.status) === "20") {
        candidates.push(row);
      }
    }

    const total = result.recordsFiltered || result.total || 0;
    if (page * pageSize >= total) {
      break;
    }

    page += 1;
  }

  if (dryRun) {
    return {
      mode: "dryRun",
      count: candidates.length,
      items: candidates
    };
  }

  const results = [];

  for (const item of candidates) {
    const latest = await queryWkReportById(item.wkId);

    if (!latest || String(latest.status) !== "20" || latest.prodPerson !== userId) {
      results.push({
        wkId: item.wkId,
        status: "skipped",
        reason: "状态或负责人已变化"
      });
      continue;
    }

    const approveResult = await addReply({
      wkId: item.wkId,
      reply
    });

    const verified = await queryWkReportById(item.wkId);

    results.push({
      wkId: item.wkId,
      wkName: item.wkName,
      status: verified && String(verified.status) === "30" ? "success" : "unknown",
      response: approveResult,
      latestStatus: verified?.status,
      replySummary: verified?.replySummary,
      approvalTime: verified?.approvalTime
    });
  }

  return results;
}
```

## 5. 批量审核接口封装建议

如果要在业务系统中新增一个真正的批量审核后端接口，建议由后端封装现有单条批复能力。

### 5.1 建议接口

```http
POST /jxpmo/rest/project/WkReportService/batchAddReply
Content-Type: application/json
```

### 5.2 请求体

```json
{
  "year": 2026,
  "month": 5,
  "reply": "已核实",
  "wkIds": [
    "f55a2a2d01284285afe157f2dc454d2f",
    "af3c9d66cf6c4656a61a8ad2f589760a"
  ]
}
```

### 5.3 服务端校验规则

后端不能直接信任前端传入的 `wkIds`，必须逐条校验：

```text
1. 当前登录用户有效。
2. 周报存在。
3. 周报 prodPerson == 当前用户 userId。
4. 周报 status == "20"。
5. 批复意见 reply 非空。
6. 已批复、草稿、非本人负责的数据全部跳过或返回失败原因。
```

### 5.4 响应建议

```json
{
  "total": 2,
  "success": 1,
  "failed": 1,
  "items": [
    {
      "wkId": "f55a2a2d01284285afe157f2dc454d2f",
      "result": "success",
      "message": "批复完成",
      "status": "30"
    },
    {
      "wkId": "af3c9d66cf6c4656a61a8ad2f589760a",
      "result": "failed",
      "message": "周报状态已变化，不是待审核"
    }
  ]
}
```

## 6. 前端批量审核交互建议

### 6.1 页面操作

建议前端增加：

- `批量审核` 按钮。
- 统一批复意见输入框。
- 待审核周报预览列表。
- 二次确认弹窗。
- 批量结果反馈表。

### 6.2 表格筛选

前端列表中应只允许勾选满足以下条件的数据：

```text
prodPerson == 当前用户 userId
AND status == "20"
```

不满足条件的数据应禁用勾选，并提示原因：

| 情况 | 提示 |
|---|---|
| `status == "10"` | 草稿周报不能批复 |
| `status == "30"` | 周报已批复 |
| `prodPerson != 当前用户 userId` | 非当前用户负责，不能批复 |

## 7. 异常处理

### 7.1 常见异常

| 场景 | 处理方式 |
|---|---|
| 当前用户接口返回 401 或登录页 | 提示登录失效 |
| 查询列表为空 | 提示当前月份无待审核周报 |
| 批复接口返回非 `批复完成` | 标记该条失败，继续处理下一条 |
| 批复成功但复查状态不是 `30` | 标记为待确认 |
| 某条周报批复失败 | 不影响其他周报继续批复 |

### 7.2 幂等处理

批量审核应按以下规则避免重复批复：

```text
提交前：只处理 status == "20" 的记录。
提交后：复查 status == "30" 且 replySummary == 本次批复意见。
重复执行：已是 status == "30" 的记录直接跳过。
```

### 7.3 并发建议

当前抓包中的批复接口是单条 GET 请求。为降低误操作和服务端压力，建议：

- 默认串行提交。
- 如需并发，限制并发数，例如 3 到 5。
- 每条记录单独记录结果。
- 不要因为某条失败中断整个批次。

## 8. 安全注意事项

- `addReply` 使用 GET 修改数据，调用方必须显式触发，不能被预加载、缓存或爬虫触发。
- `reply` 在 URL 中传输，日志系统可能记录批复意见；如可改造，建议后端新增 POST 接口。
- 批量审核必须由服务端校验负责人和状态，不能只依赖前端过滤。
- 不要在前端或日志中输出完整 Cookie、Token、手机号、邮箱等敏感信息。

## 9. 最小可用实现方案

如果不改后端，只做前端或脚本增强，最小方案如下：

```text
1. GET /jxpmo/rest/org/user
2. GET /jxpmo/rest/project/WkReportService/query?...&likeAll=当前用户姓名
3. 本地筛选 prodPerson == 当前用户 userId 且 status == "20"
4. 循环 GET /jxpmo/rest/project/WkReportService/addReply?format=json&wkId=...&reply=...
5. 对每个 wkId 调用 queryByProjectInfo 复查 status == "30"
```

如果允许后端改造，推荐新增 `POST batchAddReply`，由后端完成筛选、校验、循环批复和结果汇总。
